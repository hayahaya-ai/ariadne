# v0.2.0-rc.1 public repository calibration

Date: 2026-07-10  
Release: `v0.2.0-rc.1`  
Release commit: `fda2765bc9575034f8abd8afcb268df07b98e85e`

## Method

The published Darwin arm64 release artifact was downloaded into a disposable
workspace and checked against the release's `SHA256SUMS.txt`. Eight public
repositories were shallow-cloned at the exact commits below. No repository
code, hook, dependency installer, test, or workflow was executed.

For every target the release binary emitted:

- the default text verdict;
- an `ariadne.verdict/v2` JSON verdict;
- a `--gate` result;
- one record in consolidated JSON and JSONL fleet scans.

Every non-clean result was then checked against the cited configuration source.
External reusable actions/workflows were read from their public GitHub source
when their semantics affected the judgment. These classifications evaluate
Ariadne's evidence and wording; they are not vulnerability disclosures about
the sampled repositories.

## Confirmed product gaps

### 1. Runtime discovery is not evidence-sensitive

The surface registry treats every `.vscode/settings.json` as a Copilot runtime,
even when the file contains only unrelated editor settings. Runtime discovery
must require a supported Copilot/agent key or a dedicated Copilot surface. Its
inverse must prove that a real Copilot setting remains discoverable.

### 2. `hardened` can be emitted without a hardened control

Verdict precedence currently returns `hardened` for any discovered runtime after
reckless/trade-off/parser-failure checks, even when `hardened[]` is empty and an
exposure remains inconclusive. This contradicts the North Star definition of
hardened as enforced controls working in the user's favor. Changing this word
is a load-bearing semantic decision and must be explicit.

### 3. Valid YAML can fail closed as malformed

The dependency-free GitHub workflow parser validates bracket balance one line
at a time. Valid flow sequences spread across lines therefore fail before any
workflow facts are collected. The correction must accept balanced multiline
flow collections while its paired inverse keeps unterminated collections
malformed.

### 4. Agent instructions can influence the wrong runtime

Generic repository instructions currently gain graph influence edges to GitHub
Actions and GitLab CI. Ordinary workflows do not interpret `AGENTS.md`,
`CLAUDE.md`, or Copilot instruction files. Trust inputs need occurrence-level
runtime applicability just as controls and authorities do.

### 5. Repository write is modeled as private-data read

GitHub write permissions create `authority:broad-local`, and both broad-local
and repository-write authorities are accepted as the private-data leg of the
data-egress family. This makes issue/PR label authority reach every root-scoped
secret-like file even when the workflow never checks out or reads it.
Repository integrity is a sensitive boundary, but it is not private data and
must not be reported as data exfiltration.

### 6. Workflow supply-chain findings use the wrong remedy

The two defensible hardening signals in this sample are mutable workflow/action
references carrying secret or write authority. They should point to immutable
action/workflow pinning and least privilege. They should not be presented as
proof that external communication itself is the missing barrier.

## Fixture-first correction order

1. Add the full paired matrix before production changes:
   - generic VS Code settings / actual Copilot settings;
   - valid multiline YAML collection / unterminated collection;
   - AGENTS plus ordinary CI / an explicitly agent-driven workflow with matching runtime;
   - write-only PR/issue workflow / same-source secret-read plus egress workflow;
   - repository-integrity authority / actual credential-access authority;
   - runtime with no enforced posture evidence / runtime with a real enforced control.
2. Record the expanded evaluator and contract tests red.
3. Fix runtime discovery and YAML parsing first.
4. Fix runtime-connected influence and boundary/authority semantics.
5. Decide and document the `hardened` precedence correction explicitly.
6. Re-run this exact pinned corpus plus the expanded benchmark.
7. Publish a new release candidate only after both are green.

## Stable-release decision

`v0.2.0-rc.1` remains useful as a calibration artifact, but this sweep does not
support promoting it unchanged to stable `v0.2.0`. The safe next milestone is a
fixture-first correctness patch and a second public sweep.
